## Why FK re-pointing must ship WITH the writer migration (sequencing)

`foreign_keys` is **ON by default** in better-sqlite3 (verified empirically twice against the repo's own driver — do NOT trust code-reading claims that it's off). A ps-only parent id (post-writer-migration) referenced by a child INSERT whose FK still targets rp(id) **fails outright**. Conversely, re-pointed FKs are valid immediately because ps ⊇ rp ids (B1's fact 1). Deploy order per host is `git pull → npm run init-db → restart` — init-db rebuilds the FKs before any new writer code runs. Bundle tables rebuild at bundle-server boot (gateway restart), also before any new writer call can arrive.

## Part 1 — FK re-pointing (9 tables)

### 1a. Main schema (`scripts/init-db.js`), 3 tables

| Table | FK today | Semantics kept |
|---|---|---|
| `research_sources` (:113) | `project_id → rp(id) ON DELETE SET NULL` | SET NULL → ps(id) |
| `research_notes` (:490, :491) | `project_id → rp(id) ON DELETE SET NULL` (+ `source_id → research_sources` — PRESERVE untouched) | SET NULL → ps(id) |
| `data_backends` (:137) | `project_id → rp(id) ON DELETE CASCADE` | CASCADE → ps(id) |

SQLite cannot ALTER an FK → **table rebuild**, as an idempotent init-db migration (one function, runs every init-db). Round-1 review verified the core mechanics empirically (external-content FTS survives DROP + RENAME-into-name with preserved rowids; `foreign_keys=OFF` prevents REFERENCES rewriting in sibling DDL; PRAGMA-inside-transaction is a no-op; the keeper connection is irrelevant) and found FOUR data-integrity traps, all now mandatory steps:

1. Detect: `PRAGMA foreign_key_list(<table>)` — skip the table if no entry references `research_projects` (idempotency).
2. `PRAGMA foreign_keys = OFF` — **outside any transaction**; restore `= ON` after.
3. **Snapshot BEFORE touching anything** (round-1 C2/C3): the table's complete index DDL from `sqlite_master` (`SELECT name, sql FROM sqlite_master WHERE type='index' AND tbl_name=? AND sql IS NOT NULL`) and its full `PRAGMA foreign_key_list` — these are the recreate/verify sources. **Never hand-list indexes** (the live tables carry UNIQUE uuid indexes from `addUuidColumn`, `init-db.js:188` — losing `idx_research_sources_uuid`/`idx_research_notes_uuid`/`idx_data_backends_uuid` even temporarily opens a uuid-duplicate window on the cross-peer identity key), and **never derive columns from `PRAGMA table_info` alone** (it does not carry REFERENCES — `research_sources.backend_id REFERENCES data_backends(id) ON DELETE SET NULL` is added live at `init-db.js:161` and would be silently dropped; `foreign_key_check` passes trivially on a missing constraint, so the loss is invisible). New DDL = canonical CREATE block + every live extra column WITH its full definition. **The live-extras set, enumerated per-table for ALL NINE rebuilt tables (round-2 N1 — "six" mis-scoping would abort maker-lab boot fleet-wide):** `research_sources`: `backend_id INTEGER REFERENCES data_backends(id) ON DELETE SET NULL` (:161), `uuid`, `origin_instance_id` (:194); `research_notes`: `lamport_ts` (:160), `updated_at` (:773), `uuid`, `origin_instance_id`; `data_backends`: `uuid`, `origin_instance_id`; `maker_sessions`/`maker_bound_devices`/`maker_transcripts`: none; `maker_learner_settings`: `age`, `avatar` (maker-lab init-tables.js:140-141 — present on EVERY live host); `data_dashboard_items`: none; `data_case_studies`: `default_voice`, `display_order` (data-dashboard init-tables.js:63-64). Assert against `table_info` + `foreign_key_list` that nothing else exists — **abort loudly on an unknown column** (abort fires at the snapshot step, before any DDL — old schema intact, old code keeps running; the mandated dry-run catches surprises pre-deploy). The abort message must name the table + column and say "add it to the rebuild's extras list (and test fixture) or remove the column". Any `addColumnIfMissing` landing on these nine tables between spec and fleet deploy must update both.
4. **Capture `sqlite_sequence`** (round-1 C1 — empirically confirmed id-reuse corruption without this): `research_sources`, `research_notes`, `data_backends`, `maker_transcripts`, `data_dashboard_items`, `data_case_studies` are all AUTOINCREMENT. `DROP TABLE` deletes their sequence row and the INSERT-SELECT re-seeds at `max(id)` — a deleted-high-id table then REUSES ids (verified: seq 3 → 2 → next insert reused 3). AUTOINCREMENT's never-reuse contract is load-bearing (`shared_items.item_id`, audit targets, cached references). Read `seq` before DROP; after RENAME, restore `MAX(captured_seq, COALESCE(MAX(id),0))`. **Exact restore shape (round-2 N2 — empirically pinned):** the RENAME CARRIES `t_new`'s sequence row to the table's name (with the wrong, lower seq — or 0 for an empty copy), so post-RENAME a row ALWAYS exists; the restore must be `UPDATE sqlite_sequence SET seq = ? WHERE name = ?`, with an INSERT only if the UPDATE reports 0 changes. **An unconditional INSERT silently defeats the fix:** `sqlite_sequence` has no unique constraint, the duplicate row is ignored (SQLite uses the first), and the deleted id gets reused anyway — verified.
5. Per table, inside one `BEGIN IMMEDIATE; … COMMIT;` (the trigger-recreation pattern, `init-db.js:403-430`), with **explicit ROLLBACK on any error before rethrowing** (a thrown-and-abandoned open transaction holds the write lock on the shared crow.db — round-1 C5):
   - `CREATE TABLE <t>_new (…)` per step 3's DDL.
   - `INSERT INTO <t>_new (<explicit columns incl. id>) SELECT <same list> FROM <t>` (rowids preserved: `research_sources.id` is the FTS `content_rowid` and `research_notes.source_id`'s target; the `source_embeddings`/`note_embeddings` tables also key off these rowids).
   - `DROP TABLE <t>; ALTER TABLE <t>_new RENAME TO <t>;`
   - Recreate ALL snapshotted indexes (step 3).
   - Restore the sequence row (step 4).
   - **`research_sources` only:** recreate `sources_ai/ad/au` (:464-479) — `DROP TABLE` killed them; `sources_fts` itself survives (resolves `content=` by name at query time). After COMMIT: `INSERT INTO sources_fts(sources_fts) VALUES('integrity-check')`, fail loudly if it throws.
6. `PRAGMA foreign_keys = ON`, then `PRAGMA foreign_key_check(<table>)` per rebuilt table — zero rows or init-db aborts with a clear error BEFORE any new writer code runs. Verify post-rebuild `foreign_key_list` matches the step-3 snapshot with only the rp→ps substitution.
7. **Placement** (round-1 S2): the rebuild runs AFTER `research_notes.updated_at` (:773), after `addUuidColumn` (:194), and after `migrateLegacyProjectsToSpaces` (:349 — guarantees ps ⊇ rp before the check). **The canonical CREATE blocks (:113, :137, :490-491) are ALSO updated to `project_spaces(id)`** so fresh installs never rebuild. Forward-reference legality verified empirically (round-2 N3): research_sources CREATE at :86 precedes project_spaces at :214; CREATE with a dangling FK succeeds even with `foreign_keys=ON` (precedent: project_spaces itself forward-references `contacts(id)`); INSERTs fail until the parent exists — and the only statements between the CREATEs are `addUuidColumn`'s UPDATEs at :185, which are safe because an UPDATE that never touches `project_id` does not resolve the dangling reference (verified). **Guard note for future edits:** any seed-INSERT or project_id-touching statement placed between :86 and :214 breaks fresh installs.

### 1c. Behavior note (documented, intended)

## Risk notes

- The rebuilds are the riskiest DDL of the overhaul: rowid preservation is mandatory (FTS content_rowid + note→source FK); explicit column lists everywhere; `PRAGMA foreign_keys` toggled OUTSIDE transactions; one transaction per table so a crash leaves either the old or the new table, never half.
- Gateways may be RUNNING during a fleet host's init-db (busy_timeout 30s is set); `BEGIN IMMEDIATE` serializes against writers. Keep each table's transaction small.
- Mixed-version window (host pulled but init-db not yet run → old FKs + new writer code): the per-host deploy script order prevents it; a human skipping init-db gets hard FK errors on creates (loud, not silent corruption) — acceptable failure direction, called out in the deploy notes.
- `updateProjectSpaceMeta` gains `type` (additive); no existing caller passes it. **Deliberate behavior change (round-1 Q2):** it throws on an empty name where the old rp UPDATE accepted `""` — `crow_update_project` catches and returns the message as an honest tool error (an empty project name was never useful; this is an improvement, not an accident).
- **workspace_dir semantics (round-1 Q1, decided):** helper-created rows record a computed `workspace_dir` that is NOT mkdir'd — this is the established behavior of every bot-board space since Stage A (directories materialize on first real use; the init-db backstop and clone path mkdir when THEY need it). Learner profiles getting slugs/workspace_dir values is accepted cosmetic surplus; a "lightweight" helper mode is YAGNI until something breaks on it.
- **Stale-code window (round-1 Q3, decided):** the per-host deploy already restarts both same-tree gateways together (different DBs: `~/.crow` vs `~/.crow-mpa` — there is no shared-DB stagger on crow). Grackle's untouchable finance gateway can hold old-code MCP children against the main crow.db across the window — harmless: old-code writers write rp, the (kept) triggers mirror to ps, migrated readers see everything. The only stale-code wrinkle is an old maker delete against a ps-only learner (orphaned ps row, settings gone) — transient, self-healing on re-delete; accepted.
- rp stays writable by the dormant trigger path and keeps mirroring; B3 owns retirement.
