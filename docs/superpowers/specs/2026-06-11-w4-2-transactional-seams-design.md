## A. gracefulShutdown — drain, then exit (index.js:542-555)

Today: SIGTERM → `Promise.allSettled([sessionManager.closeAll(), shutdownAll()])` raced vs 10s → `process.exit(0)`. The HTTP listener is never closed: in-flight requests are severed mid-response, SSE sockets die abruptly, no WAL checkpoint.

**Test-cost note (S8):** no existing test spawns the full gateway as a child process; the shutdown test needs an init-db'd temp CROW_DATA_DIR, a clean env, generous CI margins, and a headless-boot validation BEFORE writing assertions. Budget for iteration.

New sequence (all bounded; total ≤ ~15s, well under systemd's default TimeoutStopSec=90):
1. `shuttingDown = true` (existing guard). **Honest note (S1): /health is served by this same listener — after step 2 a NEW monitor poll gets ECONNREFUSED during the drain window (today the listener stays open the whole 10s race). That is the correct restart-window behavior; expect up to ~drain-window of monitor-visible refusals.**
2. `server.close()` — stop accepting; existing keep-alive idle sockets released via `server.closeIdleConnections()`.
3. Wait up to `CROW_SHUTDOWN_DRAIN_MS` (default **3000** — S2: `server.close()`'s callback fires only when ALL connections end and SSE sockets count, so any open dashboard burns the FULL window on every restart; keep it short) for in-flight requests to finish (await the close callback raced vs the timeout).
4. `server.closeAllConnections()` — sever stragglers (SSE/turbo-stream sockets are legitimately long-lived; they are *expected* casualties after the drain window — Turbo/EventSource clients auto-reconnect).
5. Existing `sessionManager.closeAll()` + `shutdownAll()` race vs 10s — UNCHANGED semantics, now after the HTTP drain.
6. Best-effort WAL checkpoint: `PRAGMA wal_checkpoint(TRUNCATE)` on the gateway's main db client, try/caught. (Probed: on a DELETE-journal DB the pragma is NOT an error — returns `{busy:0,log:-1,checkpointed:-1}` silently; the try/catch is for busy-fail when other processes share the db.) Belt-and-braces, not correctness-critical (WAL is crash-safe regardless).
7. `process.exit(0)`.

`server` is in scope (`index.js:528` region); the W4-3 split deliberately left gracefulShutdown in index.js for this work. SIGINT keeps identical handling. **Tests:** a new `tests/gateway-shutdown.test.js` — boots a disposable gateway as a child process (clean env, temp CROW_DATA_DIR, random port), opens (a) a slow in-flight request (an endpoint that responds after ~1s — use a real route or /health + artificial concurrency; if no slow route exists, validate drain via an SSE connection instead), (b) an SSE connection, sends SIGTERM, asserts: exit code 0, exit happens within the cap, the in-flight request COMPLETED with a full response, the process is gone. Keep the test tolerant of timing (CI-grade margins, ±).

## B. Queued-clone re-delivery (sharing)

Root cause: `shared_items` stores no `mode`; the boot.js pending path re-sends from a generic tableMap (a bare project row — not a bundle, no mode), so the receiver's `payload.share_type === "project" && payload.mode === "clone"` gate (boot.js:449) never matches; the queued clone lands as a no-op inbox marker (item_id 0 class).

Fix (sender-side only; receiver path UNCHANGED):
1. **Schema (additive):** `shared_items.mode TEXT` (NULL default) via `addColumnIfMissing` in `scripts/init-db.js` placed AFTER the `initTable("shared_items")` block (:576-592) — avoids the fresh-DB ordering trap; do NOT add mode to the :550 rebuild-migration CREATE (its positional `SELECT *` runs before mode exists on first new-code run). **PLUS a runtime `ensureColumn(db,'shared_items','mode','TEXT')` at sharing startup (S4 — precedent `servers/gateway/migrations.js:152`, helper `servers/db.js:136`): a host that pulls code before init-db must not throw on every clone share.**
2. **Queue time:** the clone branch's INSERT (`servers/sharing/tools/share-inbox.js:125` — verified the clone branch; :253 is the generic) writes `mode='clone'`. **ALSO FIX (S3, in scope — same bug family): the clone branch precomputes delivery_status from isConnected() BEFORE the send (:124-131); a throwing send flips only the local variable and the row stays 'delivered' — never re-delivered. Change to: INSERT 'pending', UPDATE to 'delivered' only after a successful send** (mirrors the plain branch's compensating re-mark at :277-281, which is NOT revoke-related — S7 correction — and never touches clone rows since the clone branch returns at :199).
3. **Re-delivery time (boot.js pending loop):** for rows with `mode === 'clone'` AND `share_type === 'project'`:
   - **Explicit archived check first (S5):** `SELECT archived_at FROM project_spaces WHERE id=?` — `buildProjectCloneBundle` does NOT filter archived (clone-bundle.js:38-43) and ps rows soft-delete, so it must be checked here. Archived/missing ⇒ mark `delivery_status='failed'`, `console.warn` it (S5: 'failed' is terminal and surfaced nowhere — the log line is the only operator signal), continue loop.
   - **Member-row lookup WITHOUT the revoked filter, then branch (C2):** `SELECT role, capabilities, revoked_at FROM project_members WHERE project_id=? AND contact_id=? AND mode='clone' ORDER BY id DESC LIMIT 1`. If `revoked_at IS NOT NULL` ⇒ skip the send, mark `delivery_status='failed'` + warn (NOT a new 'revoked' status — **C1: the live CHECK constraint `delivery_status IN ('pending','delivered','failed')` is baked into every fleet DB at init-db.js:557/:584 and SQLite can't ALTER a CHECK; 'revoked' would throw, get swallowed by the loop's try/catch, and the row would retry forever**). If the row is ABSENT ⇒ **deliver anyway with role 'viewer' / capabilities null** (C2: the member-row write at share time is explicitly non-fatal — share-inbox.js:153-156 — so absent-member queued clones are LEGITIMATE; skipping them would silently drop a share the tool confirmed as queued). Note: normal `crow_revoke_access` DELETEs the shared_items row outright (sharing-admin.js:72-76), so the revoked branch is defensive, mostly-dead code — keep it anyway.
   - **Rebuild the bundle FRESH via `buildProjectCloneBundle(item_id)`** (same snapshot-at-delivery semantics as live send). boot.js gains it via the helpers param (server.js already builds both from `createCloneBundleHelpers(ctx)`).
   - Send the SAME message shape as live send (`type:"share", share_type:"project", mode:"clone", payload: bundle, role, capabilities, sender, timestamp`) — receiver gate (boot.js:449) verified to match; onPeerData ignores role/capabilities (verified). Mark 'delivered' on success.
4. **Plain rows (mode NULL):** the existing tableMap path — UNCHANGED. Mixed-fleet verified safe both directions (old sender + new DB ⇒ NULL mode ⇒ plain path).
5. **Tests** (`tests/clone-redelivery.test.js`, in-process, temp dir + REAL `scripts/init-db.js` + the `execute()`-shape db wrapper — precedent `tests/project-clone-apply.test.js`; **the db is NEVER faked (W2-5B2 lesson), only peerManager is** — a real init-db DB would have caught the C1 CHECK violation): (a) queued clone row + member row → send called with mode:"clone" + full bundle + role/caps; (b) member row revoked_at set → no send, row 'failed'; (c) member row ABSENT → send WITH role viewer/null caps; (d) archived project → row 'failed', loop continues to next row; (e) plain share row → legacy tableMap path unchanged; (f) live-send-throws → row stays 'pending' (the S3 fix). Extraction: `deliverPendingShares({ db, peerManager, contact, identityCrowId, buildProjectCloneBundle })` exported from boot.js (or sibling); the onPeerConnected callback calls it — **with an OUTER try/catch so it can never reject (S6: peer-manager.js:266 invokes onPeerConnected fire-and-forget with no .catch — a rejection becomes an unhandled rejection → process crash)**. The extraction is behavior-preserving for the plain path.
