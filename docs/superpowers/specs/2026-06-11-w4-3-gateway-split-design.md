## Risks & guards

- **Mount-order drift** — the route-stack fingerprint is ordered; any reorder shows.
- **`let` → closure capture changes:** `_setupCallsSignaling` is assigned inside the calls block and read in the listen callback — module 6 must return it (or an accessor); verify the gate checks this handoff specifically.
- **Top-level await (S2 — scope corrected):** the await-every-call rule applies to MODULES 1-6 ONLY (their awaits are sequential top-level; awaited wrappers preserve timing exactly). **Inside `runPostListenSetup` (module 7), every fire-and-forget construct stays VERBATIM** — the tailnet-sync IIFE :1307, the `import().then` chains (:1345/:1383/:1427), `initProxyServers().catch` :1402 (deliberately non-blocking so health checks pass during startup), `startAutoUpdate().catch` :1420, `startScheduler().catch` :1434, crosspost/orchestrator/instance-registry blocks (:1521/:1528/:1541). Converting ANY of these to awaits serializes boot-time background work and changes startup semantics. The express listen callback itself cannot await runPostListenSetup — call it fire-and-forget exactly like the body it wraps.
- **Error-handling semantics:** each feature block has its own try/catch with console.warn — keep per-block catches verbatim inside the modules (do NOT consolidate into one try/catch around the call).
- **Gates:** spec-compliance + code-quality (sonnet) once at slice end (7 commits, mechanical); opus holistic at slice end (it's the gateway — everything routes through here). `/security-review` only if boundary code moved (it must not).
