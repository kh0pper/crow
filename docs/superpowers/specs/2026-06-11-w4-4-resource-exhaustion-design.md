## 1. Hypercore feed lifecycle (close-on-revoke, not LRU)

The leak: `sync.js:32-33` (`outFeeds`/`inFeeds` per contact) and `instance-sync.js:176-177` (per instance) only ever close feeds in process-teardown `destroy()`. Contact blocking/removal and instance revocation leave live Hypercores (≈2-4 FDs each) open until restart. A blanket LRU is WRONG here (eviction races replication); the correct shape is lifecycle closure:

- `sync.js`: add `async closeContactFeeds(contactId)` — `await feed.close()` both feeds (try/caught per feed), delete the Map entries. Hypercore close→reopen is safe (on-disk storage persists; next `initContact` reopens).
- `instance-sync.js`: add `async closeInstanceFeeds(instanceId)` — same shape.
- Call sites (review C1 — contact paths EXIST, resolved): `crow_revoke_instance` (`tools/instances.js:265-311`; `instanceSyncManager` is on fullCtx — destructure it at :14). Contact BLOCK handlers: `dashboard/panels/contacts/api-handlers.js:20-26` (`UPDATE contacts SET is_blocked = 1`) and `dashboard/panels/messages/api-handlers.js:62` — wire `closeContactFeeds(contactId)` + `peerManager.leaveContact(crowId)` into BOTH (panels reach sharing in-process; precedent `panels/skills.js:17` imports from sharing; `managers.js getManagersOrNull()` exposes syncManager/peerManager). The manual-contact DELETE path (`contacts/api-handlers.js:98-104`) CANNOT leak feeds (manual contacts have no pubkey → no feeds) — leave it.
- **S2 (race):** implement both close helpers by chaining through the existing `_initLocks` tail (instance-sync.js:284-298; same pattern in sync.js:51-62) so close can't interleave with a concurrent init.
- **S1 (documented asymmetry):** UNBLOCKING a contact does not re-init feeds (no lazy contact re-init path exists; restart or re-invite covers it) — document with a comment, do NOT build re-init. Instances DO lazily re-init on un-revoke (boot.js:379-403 / tailnet-sync paths gate on status) — add a code comment noting it.

## 5. WAL systemic (servers/db.js)

Current: `createDbClient` (db.js:224-240) try/catches `journal_mode` and WARNS PER CONNECTION — the boot storm prints hundreds of `[db] Failed to set journal_mode: database is locked` lines (live crow logs confirm). `ensureKeeper` (:205-212) swallows silently and registers the keeper regardless. `resolveJournalMode` (:199-203) is env-only, default WAL.

- **(a) READ-FIRST, deduped, non-fatal mode handling (review C4 — empirically corrected):** the pragma BOTH returns-the-resulting-mode on a declined switch AND **throws `SQLITE_BUSY` under cross-process lock contention — and better-sqlite3's default `busy_timeout` is 5000ms (probed), so each failed flip attempt synchronously BLOCKS THE EVENT LOOP up to 5s per connection during a boot storm. The current spam is also a serialized multi-second boot stall.** Design: **READ the current mode first** (`db.pragma('journal_mode', {simple:true})` — read-only, lock-free); only attempt the flip when it differs from the requested mode; keep the try/catch on the flip; on catch OR mismatch, re-read and dedup-log ONE line per (filePath, requestedMode) per process saying what mode the DB is actually in, and continue. Memoize failed (path, requestedMode) attempts per process so later connections skip the flip attempt entirely (no repeat stalls). NO automatic force-DELETE fallback — switching modes needs exclusivity and a surprise switch on a shared live DB is worse than staying put. (Refines the handoff's "DELETE fallback" wording: the fallback is *accept-current-mode + log*, not *force-DELETE*.)
- **(b) Auto-select DELETE on low-RAM hosts:** in `resolveJournalMode`, when `CROW_JOURNAL_MODE` is UNSET: probe `os.totalmem()`; if ≤ `CROW_WAL_MIN_RAM_GB` (default 2) GiB → return "DELETE" and log once (a nominal-2GB host reports slightly under 2GiB — `<=` catches it; 4GB hosts ≈3.8GiB → WAL). Env override always wins (black-swan's drop-in keeps working). Probe injectable for tests. **Dedup-log ALL THREE new log lines: mode-mismatch per (path,mode), the auto-select decision, keeper-skip per path.**
- **(c) Keeper hygiene (the deleted-wal-fd class):** keeper registration is **result-read-based, never exception-based**: after the (read-first) mode resolution, register the keeper ONLY when the post-attempt READ shows `wal` (a transient locked error on a genuinely-WAL DB still registers — the read reflects the true mode regardless of who set it). Run/re-check `ensureKeeper` AFTER the client's own pragma resolution inside the same synchronous `createDbClient` body (currently keeper-first at :227; nothing depends on that order) to close the read-then-flip race. On non-WAL DBs: close + don't register (saves the fd; removes the keeper-holding-deleted-wal-fd scenario). **C5 (MANDATORY): `performBackup` (db.js:276-281) does `_dbKeepers.get(filePath).backup(...)` — keeper-skip would TypeError the admin backup route on exactly the DELETE-pinned hosts. Add a transient-handle fallback (open, backup, close) when no keeper is registered — harmless on a DELETE-mode DB (the never-open-externally hazard in its doc comment is WAL-specific).**
- Tests: `tests/db-journal-mode.test.js` — temp DBs: (i) WAL host: keeper registered, client in WAL; (ii) simulated low-RAM (injected probe): resolveJournalMode returns DELETE, keeper NOT registered, client works in DELETE; (iii) dedup: two clients on the same path under a forced mode mismatch → one warn line (capture console).
