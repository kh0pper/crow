## Why readers-first is safe (the load-bearing facts — two were CORRECTED by plan review)

1. **id continuity:** `project_spaces.id` mirrors `research_projects.id` for every trigger/backstop-created row (`tr_rp_to_ps_ins` inserts with `NEW.id`, `init-db.js:411`; `migrateLegacyProjectsToSpaces` copies ids, `:324`). All FKs (`research_sources/research_notes/data_backends/maker-lab children .project_id|learner_id`) stay valid under ps reads.
2. **ps ⊇ rp — but only with the id-collision fix below (review C2).** rp uses `INTEGER PRIMARY KEY AUTOINCREMENT` (own `sqlite_sequence`, live seq = 12); ps uses plain rowid allocation. Two ps-only writers are live TODAY (bot-board create `bot-board-api.js:489`; clone receive `sharing/server.js:855→:1255`). A ps-only create takes id 13 → the next legacy rp INSERT also takes 13 → `tr_rp_to_ps_ins` is `INSERT OR IGNORE` (`:409`) and silently skips → the backstop's `LEFT JOIN … WHERE ps.id IS NULL` (`:301-302`) finds the WRONG row at that id and cannot heal. Post-B1 that legacy project would be **invisible to every migrated reader and its sources/notes mis-attributed** — reads as data loss. **Fix (in scope):** `createProjectSpace` keeps rp's sequence ahead of ps inside its batch, with two implementation pins (round-2 review — both traps verified empirically and INVISIBLE to the tests):
   - **Seed SQL must be** `INSERT INTO sqlite_sequence(name, seq) SELECT 'research_projects', 0 WHERE NOT EXISTS (SELECT 1 FROM sqlite_sequence WHERE name = 'research_projects')` — explicitly NOT `INSERT OR IGNORE`: `sqlite_sequence` has no unique constraint on `name`, so OR IGNORE inserts a duplicate row per call (unbounded junk + formally undefined AUTOINCREMENT behavior with duplicate sequence rows — and the broken form happens to pass test 2).
   - Then `UPDATE sqlite_sequence SET seq = MAX(seq, COALESCE((SELECT MAX(id) FROM project_spaces), 0)) WHERE name = 'research_projects'` (COALESCE is free insurance — `MAX(seq, NULL)` would NULL the high-water mark).
   - **The guard pair must be the LAST statements of the default-mode batch:** the seed INSERT resets `last_insert_rowid()` (to the sqlite_sequence row's own rowid), so placing it before the slug-finalize UPDATE (`project-spaces.js:135`) or the member-row INSERT (`:143`) — both of which consume `last_insert_rowid()` — would silently corrupt the new space. Explicit-slug mode's member row uses a JS-bound id post-batch, but keep the guard last there too for uniformity.
   (ps's own next rowid is always > max(rp.id) because ps ⊇ rp ids — the collision only ever runs in the rp→ps direction.) Live DBs verified collision-armed (rp seq 12, ps max 12) but not yet collided (rp ≡ ps).
3. **Read-ps / write-rp is coherent for INSERT/UPDATE; DELETE ARCHIVES (review C1 — the critical correction).** SQLite triggers run inside the triggering statement (same transaction), and `tr_rp_to_ps_upd` (`:436-441`) mirrors every reader-used column (name, description, type, status, tags, updated_at). But `tr_rp_to_ps_del` (`:445-452`) does NOT delete — it sets `archived_at`, leaving `type`/`status` untouched. A naive mechanical rename would resurrect every rp-deleted row in migrated readers: deleted learners reappearing in maker-lab lists and the panel (a **privacy regression** — `maker_delete_learner` is the parental-deletion flow), a single ever-deleted learner permanently blocking `maker_set_mode` solo-downgrade counts, and `ensureDefaultLearner` returning a ghost id whose child-table FK inserts then violate `REFERENCES research_projects(id)`. **Fix (in scope): every migrated read adds `archived_at IS NULL`** — in WHERE clauses, and in LEFT JOIN ON-conditions (`LEFT JOIN project_spaces p ON … AND p.archived_at IS NULL`) so name-decoration JOINs keep returning NULL for deleted projects exactly as a missing rp row did. This is a deliberate, documented deviation from "filters unchanged". (The archive-on-delete trigger semantic itself is intentional Phase-1 design — spaces are recoverable — and stays. **B2 note:** when maker-lab's delete moves to ps-direct, learner_profile deletion must HARD-delete, and the PII already retained in archived ps rows from any past learner deletions must be purged.)
4. **Column compatibility:** rp's columns are a strict subset of ps; every migrated query references only subset columns. Two value-level deltas, accepted + documented: (a) ps's `type` is `NOT NULL DEFAULT 'general'` and the ins trigger coalesces NULL→`'general'` (`:416`), so a NULL-type rp row reads back as `'general'` — `COALESCE(type,'research')` displays (`research/server.js:615`, `:191`) shift from "research" to "general" for such rows (cosmetic); (b) `uuid` can diverge between rp and ps for legacy rows (trigger mints one, init-db backfills rp with a different one, upd trigger never mirrors it) — nothing in B1 consumes uuid, but B2 must NOT key sharing off it without reconciling.
