**C3 (lazy glasses WebSocket) — DECISION PENDING (likely DEFER to its own wave).**
Round-1 review found the plan's C3 design materially wrong and risky:
- **The real per-device FGS auto-start is in `CrowApplication.kt:25-41`, NOT BootReceiver.**
  It loops paired devices and `startForegroundService(GlassesService)` on EVERY process
  create (launcher tap, boot, tile, anything). Removing only the BootReceiver loop does
  NOT stop "one FGS per device" — CrowApplication still does it. And removing the bare
  `onStartCommand` connect (the C3 core edit) without handling CrowApplication leaves an
  orphan "Connecting to glasses…" foreground notification that never connects/self-stops —
  a user-visible regression. The original plan never mentioned CrowApplication.
- **The "tie to BT-connection" fix is feasible but non-trivial:** DAT 0.5.0 exposes
  `Wearables.devicesMetadata[id].linkState` (StateFlow<DeviceMetadata> →
  LinkState{DISCONNECTED,CONNECTING,CONNECTED}) — a Kotlin coroutine collector in
  CrowApplication, NOT a simple ACL BroadcastReceiver (raw ACL fires for any BT device).
  All `com.meta.wearable.dat.core.internal.*` BT classes are non-callable.
- **Confirmed:** inbound `remote_turn`/`note_stream_begin`/`media_control`/`capture_photo`
  (GlassesService.java:758-790) REQUIRE an already-open socket. So the PTT-only fallback
  silently breaks every server-initiated turn/recording/photo until the user does one
  manual PTT per session — undiagnosable for a non-technical user, violating
  [[fix-the-product-not-the-instance]].
- **Review recommendation: split C3 into its own dedicated wave** with the proper
  CrowApplication/StateFlow design + an on-device server-push test plan. Ship
  A+C1+C2+C4+C5+D now (real battery + alignment wins, all code-reviewable, zero glasses
  risk). → OPERATOR DECISION below.

**C6b (GlassesService) — bundled with C3 (same reconnect machinery), so moves with the
C3 decision.** `scheduleReconnect` (851-860) retries forever silently. Add a
consecutive-failure counter in BOTH `onFailure` (838) AND `onClosed` (823) — a
gateway-restart produces a clean 1000 close that also reconnects, so onFailure alone won't
bound it. After ~6, stop scheduling, set notification "Glasses offline — reopen to retry",
`stopSelf()` if not in a turn/note-stream/media session; a fresh `ACTION_BEGIN_TURN` resets
+ reconnects.
